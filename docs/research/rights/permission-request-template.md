# Vendor MIB rights request template

This is an operational request template, not legal advice. Send it from an
account that can bind the project owner, preserve the complete reply and
attachments, and do not summarize silence as consent.

## Subject

Permission request: public MIB metadata, rendered documentation, and API use

## Message

Hello,

I operate **mibvendor**, a source-aware MIB discovery and OID-resolution web
application planned for `mibvendor.io`. I am requesting written clarification
for `[VENDOR]` MIB modules obtained from `[OFFICIAL SOURCE URL]`.

Please confirm which of the following uses `[VENDOR]` authorizes. A separate
yes/no answer for each line is important because the service will enforce the
scopes independently:

1. **Metadata index:** publicly display and search OID, symbol, module name,
   module revision, object kind, and official source link.
2. **Rendered text:** publicly display DESCRIPTION, textual conventions, enums,
   INDEX/AUGMENTS definitions, and notification/varbind documentation.
3. **API output:** return the approved fields through a free or paid JSON API.
4. **Raw download:** mirror the original MIB files for individual download.
5. **Bulk export:** provide dataset/API snapshots containing the approved
   fields.

The requested permission should cover worldwide commercial use,
normalization/derived structured data, caching/CDN delivery, and versioned
updates. Please also specify:

- required attribution and notices;
- whether modification/normalization is allowed;
- whether permission is perpetual or can expire/revoke;
- any product/account entitlement restrictions;
- the takedown/correction contact and required response procedure;
- whether the permission applies to all `[VENDOR]` MIBs or only listed modules
  and versions.

We will not collect vendor portal credentials, bypass access controls, or state
that public download alone grants redistribution permission. Until written
scope is confirmed, the material will not enter the public corpus.

Project owner: `[LEGAL NAME / ENTITY]`
Contact: `[ACCOUNTABLE EMAIL]`
Project: `https://github.com/ta2jam/mibvendor`
Planned service: `https://mibvendor.io`

Regards,

`[NAME AND ROLE]`

## Evidence to retain

- exact sent message and timestamp;
- sender identity and authority;
- recipient address/contact channel;
- full response headers or ticket/export;
- attached license or terms version;
- covered product/module/version list;
- each approved/denied scope;
- attribution, expiry, revocation, and takedown terms;
- reviewer and review date.

An account-team statement that does not clearly grant the requested scopes is
not approval. Route ambiguous replies back to the vendor's legal/licensing
contact.
